## Security Review — 2026-04-23 (scope: paths)

**Summary:** Reviewed 7 paths covering the world-admin CLI (`daydream/admin.py`), the authenticated WS endpoint and input router (`daydream/api/ws.py`), the data-skill safety primitives (`daydream/llm/safety.py`), the DB-loaded data-skill executor (`daydream/skills/data.py`), the effect allowlist dispatcher (`daydream/skills/effects.py`), the registry merge (`daydream/skills/registry.py`), and the authored forge showcase (`skills/forge.json`). Every SQL call (admin cascade DELETE, `skills` upsert, effects `INSERT items` / `UPDATE toons`, events append, toon / room reads) is parameterized. `admin.cmd_restore` uses PEP 706's `filter="data"` on `tarfile.extract` plus a belt-and-suspenders `startswith("/") or ".."` check that runs before extraction, so a hostile archive member cannot write outside `data_dir`; the manifest format-version and schema-version gates run before any payload bytes touch disk. `admin.cmd_skill_add` is an operator-only CLI, not reachable via the web layer — `bin/game world skill add` wraps `python -m daydream.admin` and nothing in `daydream/server.py` or `daydream/api/` imports from `daydream.admin`. The WS endpoint gates with `auth.is_authed` at the very top (`ws.py:241-243`); `_handle_input` then routes through `registry.list_available_for_room(room_id)` for BOTH the canonical bypass AND the interpreter fallback, so an LLM-chosen `decision.skill` that is not in the current room's candidate set is rejected by `available_by_name.get(decision.skill) is None` at `ws.py:221-223` — the interpreter cannot promote a player into a skill their room doesn't grant them. Data-skill prompt injection is contained by three layers: (1) `safety.wrap_player_input` wraps player text in `<player_input>...</player_input>` role-separator tags and neutralizes any literal closing tag so the player cannot break out of the quoted region, (2) `jinja2.sandbox.SandboxedEnvironment` (Jinja2 3.1.6) denies `__class__` / `__globals__` access from an authored template, and (3) `effects.ALLOWED_KINDS` restricts LLM-emitted effects to `narrate`, `add_item`, `set_mood`. `_apply_add_item` and `_apply_set_mood` use parameterized SQL and strip the LLM's `name` / `seed` / `mood` strings; the `toon_id` field in `set_mood` is LLM-controllable but operates entirely within the existing friend-scope trust boundary (single-player v0, explicit v1-allowlist-trusted posture documented in `effects.py` module docstring). `safety.first_banned` regex patterns use `\b(?:...)\b` alternations without nested quantifiers, so no ReDoS backtracking path. Git history for the seven files shows no secrets and no prior-state sensitive content; `data.py`, `effects.py`, and `safety.py` were born in the current data-skill slice and have no historical surface area to scan. Prior accepted risks from 2026-04-23 still apply unchanged and are out of scope for these paths.

### Findings

No security issues identified.

### Accepted Risks

- Cookie `https_only=False` in `daydream/server.py:38` is documented inline ("friend-scope; box is on a private LAN/Tailscale only").
- No CSRF token on `/api/login` and `/api/logout`. `SessionMiddleware` defaults to `SameSite=Lax`, which blocks cross-site POST from carrying the session cookie; the worst-case residual impact under the documented threat model is forced-logout from a same-site phishing page, consistent with friend-scope posture.
- The 100.64.0.0/10 hardcoding in `daydream/api/access.py:25` is correct because Tailscale's CGNAT range is fixed by their design. A self-hosted Headscale with a custom range would need that constant updated. Documented in CLAUDE.md.
- `AccessMiddleware` reads `scope["client"][0]` directly rather than honoring forwarded-for headers. This is the secure default given uvicorn is not started with `--proxy-headers`. Operators who later put the app behind a reverse proxy must add `--forwarded-allow-ips` themselves AND audit that the proxy strips inbound `X-Forwarded-For`.
- `bin/game` sources `.env` and `~/.config/daydream/secrets.env` via `set -a; source`; a writable env file is equivalent to arbitrary code execution. Operator-controlled, gitignored, standard practice.
- `bin/game cmd_logs` passes `$1` through `_log_file` without validation; an operator typing `bin/game logs ../../etc/passwd` would resolve into `$RUNDIR/../../etc/passwd.log`. Operator-invoked only; anything the operator can `tail` through this path they can already read directly on the box.
- `bin/qpeek-bootstrap` clones `https://github.com/peterzat/qpeek` and runs `pip install -e` against it. Supply-chain dependency on the project owner's own GitHub account; same trust boundary as the rest of the dev toolchain.
- `/cache/{world}/{target_kind}/{target_id}/{filename}` in `daydream/server.py:91` is intentionally unauthenticated so pre-auth `<img src>` fetches render. `AccessMiddleware` is the real gate; segment validation blocks traversal.
- Tailscale-mode `POST /api/login` at `daydream/api/auth.py:32-34` short-circuits to `authed=True` + redirect without checking the password. Deliberate: `AccessMiddleware` has already CGNAT-gated the request, so the password check would be redundant ceremony. Documented in the module docstring and in CLAUDE.md "Auth" + "Network access" sections.
- Data-skill `set_mood` effect handler (`daydream/skills/effects.py:128-151`) accepts an LLM-chosen `toon_id` without scoping it to the actor's world / room / ownership. In v0 single-player + friend-scope this sits inside the existing trust boundary — an authenticated player can already mutate state through play; the LLM is not a distinct actor against them. v2's per-effect jsonschema + target-authorization pipeline is tracked under BACKLOG `skills-authoring-and-security`. Documented in `daydream/skills/effects.py` module docstring.
