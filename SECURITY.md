## Security Review — 2026-05-07 (scope: paths)

**Summary:** Path-scoped review of the LLM-driven drift narrate landing (`220d4fc`) and the idempotent git-hook installer (`f52a424`). Files reviewed: `daydream/drift.py`, `bin/install-hooks`, `tests/conftest.py`, `tests/test_drift.py`. The drift loop now runs an LLM-composed narrate path on each tick (rendered through a `jinja2.sandbox.SandboxedEnvironment` with `npc_name`, `npc_seed`, `npc_mood`, and up-to-K memories from `memories.retrieve`), with fall-through to a mood-bucketed canned pool on `LLMUnavailable`, banlist hit, empty / non-string narrate, or any other failure (`drift.py:244-293`). The output-side banlist runs on the LLM's parsed `narrate` before emit (`drift.py:288-292`), and the capture-side banlist (`daydream/memories.py:195-198`) prevents banlist-tripping text from ever entering the memory store. The only attacker-controlled surface in the new prompt is `m.text` from previously-captured memory rows; `npc_name`, `npc_seed`, `npc_mood` come from operator-authored migration SQL (`migrations/001_initial.sql`, `006_first_npc.sql`, `008_second_npc.sql`) and are not player-mutable in v0. The drift LLM call uses `acompletion_json`, which already wraps the GPU arbiter, so no new arbiter machinery and no new bypass surface. The `_list_npcs` SELECT (`drift.py:179-187`) is a static query with no parameters; the drift loop emits with `actor_type="system"` and never performs auth-sensitive work. `bin/install-hooks` is operator-invoked, accepts only `""` / `--uninstall` / `--help` arguments (no path inputs), constructs `HOOKS_DIR` from `BASH_SOURCE[0]` (script-resident path), uses fixed-string `grep -qF` for marker detection, refuses to overwrite hooks lacking the marker, and writes only inside `.git/hooks/` — same operator-trust class as `bin/game` itself. `tests/conftest.py` adds a `DAYDREAM_DRIFT_LLM_ENABLED=0` default mirroring the existing `DAYDREAM_DRIFT_ENABLED=0` and `DAYDREAM_MEMORY_ENABLED=0` test defaults; the new line is a defensive disable, not a new attack surface. The previously-accepted hardcoded `test-password` and `test-session-secret-not-for-production` placeholders are unchanged. `tests/test_drift.py` mocks `acompletion_json` in every LLM-touching test (`AsyncMock(...)`) — no test contacts vLLM. Git history (`git log -p`) for all four files shows no secret values introduced or removed; the only "password"/"key" string occurrences in the diff are the prior-accepted test placeholders and the `monkeypatch.setenv("DAYDREAM_DRIFT_LLM_ENABLED", ...)` test infrastructure. No PII beyond commit-author metadata (out of scope per skill instructions).

The drift `_DRIFT_USER_TEMPLATE` at `drift.py:140` renders `<memory>{{ m.text }}</memory>` without neutralizing literal `</memory>` substrings inside captured text — structurally identical to the close-tag-injection class already documented in the prior review's WARN for `skills/{rook,iris}.json:6`. This is NOT a new finding: it is the same accepted-risk WARN, instantiated at a new call site, with the same backstop posture (capture-side banlist filters tone-banned text; output-side banlist filters tone-banned LLM responses; canned-pool fallback fires on any LLM failure path). The drift output shape (`{"narrate": "<sentence>"}` with no `effects` parsing) gives drift a strictly smaller blast radius than the dialogue path that already accepts this risk — there is no effect-allowlist to subvert, no `set_mood`/`give_item` machinery downstream, no auth-sensitive sink. Per the skill instructions ("Do not re-flag accepted risks as findings"), no new finding is added; the prior WARN's mitigation list (banlist-at-capture is now in place; `</memory>`-neutralization at retrieval-time and v2's `skills-authoring-and-security` per-effect jsonschema + audit log are still pending) covers the drift call site automatically.

### Accepted Risks

Carried forward from prior reviews; none re-flagged this scope:

- Cookie `https_only=False` in `daydream/server.py:42` (friend-scope; LAN/Tailscale only).
- No CSRF token on `/api/login` and `/api/logout`. `SameSite=Lax` blocks cross-site POST from carrying the session cookie; worst case under the documented threat model is forced-logout from a same-site phishing page.
- The `100.64.0.0/10` Tailscale CGNAT hardcoding in `daydream/api/access.py`. Self-hosted Headscale with a custom range would need this updated. Documented in CLAUDE.md.
- `AccessMiddleware` reads `scope["client"][0]` directly rather than honoring forwarded-for headers. Secure default given uvicorn does not start with `--proxy-headers`. Operators putting the app behind a reverse proxy must add `--forwarded-allow-ips` AND audit that the proxy strips inbound `X-Forwarded-For`.
- `bin/game` sources `.env` and `~/.config/daydream/secrets.env` via `set -a; source`; a writable env file is equivalent to arbiter code execution. Operator-controlled, gitignored, standard practice. The `DAYDREAM_DRIFT_*` (including the new `DAYDREAM_DRIFT_LLM_ENABLED` and `DAYDREAM_DRIFT_LLM_TOP_K`) and `DAYDREAM_MEMORY_*` env vars sit inside this same trust boundary; their parsers float-/int-coerce with a `ValueError` fallback to the default and never propagate to security-sensitive code paths.
- `bin/game cmd_logs` passes `$1` through `_log_file` without validation. Operator-invoked only; anything the operator can `tail` through this path they can already read directly.
- `bin/qpeek-bootstrap` clones from `github.com/peterzat/qpeek` and runs `pip install -e`. Same trust boundary as the rest of the dev toolchain.
- `/cache/{world}/{target_kind}/{target_id}/{filename}` in `daydream/server.py:95-107` is intentionally unauthenticated so pre-auth `<img src>` fetches render. `AccessMiddleware` is the real gate; segment validation blocks traversal.
- Tailscale-mode `POST /api/login` at `daydream/api/auth.py` short-circuits to `authed=True` without checking the password. `AccessMiddleware` has already CGNAT-gated the request. Documented in module docstring + CLAUDE.md.
- Data-skill `set_mood` effect handler accepts an LLM-chosen `toon_id` without scoping it. v2's per-effect jsonschema + target-authorization pipeline is tracked under BACKLOG `skills-authoring-and-security`.
- `bin/vllm-bootstrap` and `bin/memory-bootstrap` interpolate `$MODEL` (sourced from `DAYDREAM_VLLM_MODEL` / `DAYDREAM_MEMORY_MODEL`) into an unquoted heredoc that becomes Python source. Same operator-trust class as `bin/game`'s env sourcing.
- Stored prompt-injection via captured memory text bypasses `wrap_player_input` containment (`daydream/skills/data.py:362`, `daydream/memories.py:175-209`, `skills/{rook,iris}.json:6`). Capture-time banlist is now in place; `<memory>...</memory>` separator wraps the rendered text; `</memory>` substrings are NOT neutralized at retrieval time. The drift LLM path at `daydream/drift.py:140` is a second instance of the same wrapper without a `</memory>`-neutralization layer; same backstop posture (capture-side banlist + output-side banlist + canned-pool fallback). Realistic impact at v0 is bounded to mild voice/tone deviation, not auth bypass / RCE / data exfiltration. Deeper fix is v2's `skills-authoring-and-security`.
