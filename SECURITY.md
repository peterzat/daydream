## Security Review ŌĆö 2026-05-27 (scope: paths)

**Summary:** Path-scoped review of the `snapshot-restore-commands` landing (commit `89697ad`, HEAD). Two files: `daydream/admin.py` (the operator-only world admin CLI; this turn added `cmd_snapshot` / `cmd_snapshot_restore` and their argparse wiring) and `tests/test_admin.py` (11 new tier_medium snapshot tests). This is the first security pass over `admin.py` itself; the prior admin-touching review (`dd983fa`) was scoped to the bootstrap path only. Trust model: `admin.py` has no network entry point ŌĆö it is reachable solely via `bin/game world {list,archive,restore,snapshot,snapshot-restore,verify,delete,skill add,bootstrap}` from the box operator's shell. The only external inputs are operator-supplied: a `world_id`, an archive/snapshot file path, a skill JSON path. The documented adversarial surface is malformed/foreign archive or snapshot files (the "ship a world to another box" workflow), not anonymous network attackers. Traced each sink: (1) **SQL** ŌĆö every query is parameterized (`?`): the `SELECT id FROM worlds WHERE id = ?` existence checks, the `COUNT(*) ... WHERE world_id = ?` counts, the hand-rolled DELETE cascade in `cmd_delete` (events/generated_assets/memories/items/toons/rooms/worlds, all `WHERE ... = ?`), the `skills` ON CONFLICT upsert in `cmd_skill_add`, and the `assets.*` helpers (`list_assets` builds its WHERE from bound params, never f-strings). No string interpolation into SQL anywhere. (2) **Path traversal via `world_id`** ŌĆö `cmd_delete` does `shutil.rmtree(image_cache.cache_dir() / world_id)` and `cmd_snapshot`/`cmd_archive` fold `world_id` into output filenames; but every destructive/write path first gates on `SELECT id FROM worlds WHERE id = ?` returning a row, so a `../`-bearing string never matches a real world id and the command exits 2 before reaching the filesystem sink. World ids are operator/Opus-authored slugs (`w-bunny` shape). (3) **Tar extraction (`cmd_restore`)** ŌĆö explicit CVE-2007-4559 belt-and-suspenders guard rejects members starting with `/` or containing `..` (`admin.py:317`), then extracts with `filter="data"` (`admin.py:324`) as the authoritative check; `tarfile.data_filter` is present on this interpreter (Python 3.10.12, backport confirmed) so symlink/device/abspath escapes are also blocked. `test_restore_rejects_path_traversal_member` proves a `../../../tmp/...` member is refused and nothing lands outside `data_dir`. Manifest format + schema-version are validated before any payload byte is extracted (`admin.py:286-304`). (4) **Foreign-DB content (`cmd_snapshot_restore`)** ŌĆö the candidate `.db` is probed read-only + immutable (`file:...?mode=ro&immutable=1`, `admin.py:421`) so the probe creates no `-wal/-shm` sidecars and cannot mutate the file, validates it parses as SQLite and that its own `_migrations` max is not newer than `_max_known_migration()`, refuses to overwrite an existing live DB, and requires `--yes`. Installing a well-formed-but-logically-bogus world is an explicit operator choice within the same trust class as handing the app any DB file; SQLite executes no code on open, so there is no RCE path. (5) **Data exposure / logging** ŌĆö error/print output carries only operator-supplied paths, world ids, schema-version integers, and asset counts; no secrets (the project's only secret, `DAYDREAM_PASSWORD`/`session_secret`, lives in `config.py`, out of scope and already an accepted risk). `git log -p --follow` over `admin.py` surfaces no credential values, only code. Test fixtures use fictional names only ("bunny-world", "Bramble", "Wren", "Rook", "Iris", "meadow", "Foo", "other"); no PII. 34/34 tests green. Per skill instructions, no accepted risks are re-flagged here.

### Accepted Risks

Carried forward from prior reviews; none re-flagged this scope:

- Cookie `https_only=False` in `daydream/server.py:42` (friend-scope; LAN/Tailscale only).
- No CSRF token on `/api/login` and `/api/logout`. `SameSite=Lax` blocks cross-site POST from carrying the session cookie; worst case under the documented threat model is forced-logout from a same-site phishing page. The `POST /api/slots/{slot}/{create,claim,kick}` endpoints inherit the same posture (no CSRF token); same-origin-only state changes from a tailnet-resident attacker page are blocked by `SameSite=Lax` for the `daydream_session` cookie, and unauthed cross-site POSTs from outside the tailnet are 403'd by `AccessMiddleware` before reaching these handlers. Multi-user differentiation that would tighten this lands with v2.
- The `100.64.0.0/10` Tailscale CGNAT hardcoding in `daydream/api/access.py`. Self-hosted Headscale with a custom range would need this updated. Documented in CLAUDE.md.
- `AccessMiddleware` reads `scope["client"][0]` directly rather than honoring forwarded-for headers. Secure default given uvicorn does not start with `--proxy-headers`. Operators putting the app behind a reverse proxy must add `--forwarded-allow-ips` AND audit that the proxy strips inbound `X-Forwarded-For`.
- `bin/game` sources `.env` and `~/.config/daydream/secrets.env` via `set -a; source`; a writable env file is equivalent to arbiter code execution. Operator-controlled, gitignored, standard practice. The `DAYDREAM_DRIFT_*` and `DAYDREAM_MEMORY_*` env vars sit inside this same trust boundary; their parsers float-/int-coerce with a `ValueError` fallback to the default and never propagate to security-sensitive code paths.
- `bin/game cmd_logs` passes `$1` through `_log_file` without validation. Operator-invoked only.
- `bin/qpeek-bootstrap` clones from `github.com/peterzat/qpeek` and runs `pip install -e`. Same trust boundary as the rest of the dev toolchain.
- `/cache/{world}/{target_kind}/{target_id}/{filename}` in `daydream/server.py:118-130` is intentionally unauthenticated so pre-auth `<img src>` fetches render. `AccessMiddleware` is the real gate; segment validation blocks traversal.
- Tailscale-mode `POST /api/login` at `daydream/api/auth.py:46-50` short-circuits to `authed=True` without checking the password. `AccessMiddleware` has already CGNAT-gated the request. Documented in module docstring + CLAUDE.md.
- Data-skill `set_mood` effect handler accepts an LLM-chosen `toon_id` without scoping it (`daydream/skills/effects.py:128-151`). v2's per-effect jsonschema + target-authorization pipeline is tracked under BACKLOG `skills-authoring-and-security`. The `daydream.toons.set_mood` helper itself is a parameterized UPDATE; the unscoped-target risk lives at the effect-handler call site, not in the helper.
- `bin/vllm-bootstrap` and `bin/memory-bootstrap` interpolate `$MODEL` into an unquoted heredoc that becomes Python source. Same operator-trust class as `bin/game`'s env sourcing.
- Stored prompt-injection via captured memory text bypasses `wrap_player_input` containment (`daydream/skills/data.py:362`, `daydream/memories.py:175-209`, `skills/{rook,iris}.json:6`, plus the parallel drift instance at `daydream/drift.py:179-185`). Capture-time banlist is in place; `<memory>...</memory>` separator wraps the rendered text; `</memory>` substrings are NOT neutralized at retrieval time. Realistic impact at v0 is bounded to mild voice/tone deviation, not auth bypass / RCE / data exfiltration. Deeper fix is v2's `skills-authoring-and-security`.
- `/status/drift` in `daydream/server.py:60-80` is unauthenticated by design (no session check; only `AccessMiddleware` gates it). Returns three integer counters with no PII or secrets; in tailscale mode it is tailnet-only, in public mode it exposes "drift has fired N times" to any reader. Same trust class as `/login` and `/cache/...`.
- v1 friend-scope on the slot endpoints (`daydream/api/slots.py`): any auth'd session can create / claim / kick any slot ŌĆö there is no per-session ownership enforcement on `claim_slot` or `kick_slot`. Explicit in the module docstring and in SPEC 2026-05-07's v1 friend-scope callout; v2 multi-user-shared-world introduces the per-session ownership check.
- `_ensure_session_id` (`daydream/api/auth.py:29-39`) stamps a fresh `uuid.uuid4()` into `request.session` on first read in tailscale mode, even on slot-endpoint hits that did not go through `POST /api/login`. Consistent with the documented "tailnet membership IS the auth" posture; the stamped UUID is the slot picker's controller identity, not an authentication token.
- Request body size on `POST /api/slots/{slot}/create` is unbounded (no `client_max_body_size` / Starlette body-size cap). A tailnet member can POST an arbitrarily large `name` / `appearance_seed` and persist it. Same trust class as the documented friend-scope "any auth'd session can do everything" posture.
