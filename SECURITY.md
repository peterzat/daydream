### Findings

[WARN] daydream/skills/data.py:361-367, daydream/memories.py:175-209, skills/{rook,iris}.json:6 — Stored prompt-injection via captured memory text bypasses `wrap_player_input` containment.
  Attack vector: an authenticated player at the forge or attic crafts a turn-1 message that contains a benign-looking prompt-injection payload (the WHIMSY input banlist at `daydream/llm/safety.py:51-66` only filters seven tone categories — pixel-art, grimdark, sexual, violence, urgency, modern-tech, sarcasm — by word-boundary regex, NOT classic injection patterns like "ignore previous instructions", spurious system-role tags, or fake `<player_input>` envelopes). The turn-1 message *is* role-separator-wrapped via `safety.wrap_player_input` so the LLM's first call is contained. After the LLM responds and effects dispatch successfully, `daydream/skills/data.py:362` captures `f"the visitor said: {args}"` to the `memories` table — `args` is the raw, UNWRAPPED, banlist-passed-but-injection-permitting player text. On the next turn, `daydream/skills/data.py:277` retrieves the top-K and passes them to the Jinja template as the `memories` context list; the templates at `skills/rook.json:6` and `skills/iris.json:6` render each entry as `- {{ m.text }}` inside the prompt BEFORE the `<player_input>` envelope, with no role-separator and no banlist re-scan. The attacker-controlled text now sits in the prompt at NPC-perspective trust level rather than wrapped-quoted-region trust level. The data-skill output banlist (`daydream/skills/data.py:332`) and effect allowlist (`daydream/skills/effects.py`) backstop the worst cases (banned categories in narration, off-allowlist effect kinds), so realistic impact at v0 is bounded to mild voice/tone deviation in subsequent NPC narrations rather than auth bypass / RCE / data exfiltration. This is a defense-in-depth gap, not actively exploitable for high-severity outcomes given the friend-scope deployment + tailnet gating + output-side guardrails.
  Evidence: `daydream/skills/data.py:362` (`memories.capture(npc_id, world_id_for_mem, f"the visitor said: {args}")`) and `:366` (NPC narration capture). Retrieval at `:277` (`memories.retrieve(npc_id, world_id_for_mem, args)`). Template injection at `skills/rook.json:6` and `skills/iris.json:6` (`{% for m in memories %}- {{ m.text }}\n{% endfor %}` rendered before the `<player_input>` block).
  Remediation: at minimum, run `safety.first_banned` on the captured `text` AND the captured narration before persisting, mirroring the existing input/output banlist symmetry; reject the capture (return `None` + log) on hit. For the deeper class — captured memories rendered into prompts without role-separator containment — the right fix lands with v2's `skills-authoring-and-security` work (per-effect jsonschema + content-safety classifier + audit log) and is best tracked there as an additional concern rather than patched ad-hoc here. A lighter v1 mitigation: wrap memory rendering with its own `<memory>...</memory>` separator inside the templates, neutralize any literal `</memory>` in captured `text` at retrieval time, and instruct the LLM in the surrounding template language that memory blocks are quoted recall, not directives. None of these mitigations require schema or storage changes; they are template + retrieval-time edits.

### Accepted Risks

Unchanged from prior entry (carried forward; none re-flagged this scope):

- Cookie `https_only=False` in `daydream/server.py:42` is documented inline ("friend-scope; box is on a private LAN/Tailscale only").
- No CSRF token on `/api/login` and `/api/logout`. `SessionMiddleware` defaults to `SameSite=Lax`, which blocks cross-site POST from carrying the session cookie; the worst-case residual impact under the documented threat model is forced-logout from a same-site phishing page, consistent with friend-scope posture.
- The 100.64.0.0/10 hardcoding in `daydream/api/access.py` is correct because Tailscale's CGNAT range is fixed by their design. A self-hosted Headscale with a custom range would need that constant updated. Documented in CLAUDE.md.
- `AccessMiddleware` reads `scope["client"][0]` directly rather than honoring forwarded-for headers. This is the secure default given uvicorn is not started with `--proxy-headers`. Operators who later put the app behind a reverse proxy must add `--forwarded-allow-ips` themselves AND audit that the proxy strips inbound `X-Forwarded-For`.
- `bin/game` sources `.env` and `~/.config/daydream/secrets.env` via `set -a; source`; a writable env file is equivalent to arbiter code execution. Operator-controlled, gitignored, standard practice. The `DAYDREAM_DRIFT_*` env vars and the new `DAYDREAM_MEMORY_*` env vars (`DAYDREAM_MEMORY_ENABLED`, `DAYDREAM_MEMORY_MODEL`, `DAYDREAM_MEMORY_DECAY_HOURS`, `DAYDREAM_MEMORY_TOP_K`, `DAYDREAM_TORCH_CPU_INDEX`) sit inside this same trust boundary; their parsers float-/int-coerce with a `ValueError` fallback to the default and never propagate to security-sensitive code paths.
- `bin/game cmd_logs` passes `$1` through `_log_file` without validation; an operator typing `bin/game logs ../../etc/passwd` would resolve into `$RUNDIR/../../etc/passwd.log`. Operator-invoked only; anything the operator can `tail` through this path they can already read directly on the box.
- `bin/qpeek-bootstrap` clones `https://github.com/peterzat/qpeek` and runs `pip install -e` against it. Supply-chain dependency on the project owner's own GitHub account; same trust boundary as the rest of the dev toolchain.
- `/cache/{world}/{target_kind}/{target_id}/{filename}` in `daydream/server.py:95-107` is intentionally unauthenticated so pre-auth `<img src>` fetches render. `AccessMiddleware` is the real gate; segment validation blocks traversal.
- Tailscale-mode `POST /api/login` at `daydream/api/auth.py` short-circuits to `authed=True` + redirect without checking the password. Deliberate: `AccessMiddleware` has already CGNAT-gated the request, so the password check would be redundant ceremony. Documented in the module docstring and in CLAUDE.md "Auth" + "Network access" sections.
- Data-skill `set_mood` effect handler (`daydream/skills/effects.py`) accepts an LLM-chosen `toon_id` without scoping it to the actor's world / room / ownership. In v0 single-player + friend-scope this sits inside the existing trust boundary. v2's per-effect jsonschema + target-authorization pipeline is tracked under BACKLOG `skills-authoring-and-security`.
- `bin/vllm-bootstrap` interpolates `$MODEL` (sourced from `DAYDREAM_VLLM_MODEL`) into an unquoted heredoc that becomes Python source. A malicious value could escape the `snapshot_download(...)` string and run arbitrary Python in the vLLM venv. Same operator-trust class as the existing `bin/game` env-sourcing accepted risk. `bin/memory-bootstrap` follows the same pattern with `$MODEL` sourced from `DAYDREAM_MEMORY_MODEL` — same operator-trust class, same accepted-risk shape.
