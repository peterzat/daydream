## Security Review — 2026-04-23 (scope: paths)

**Summary:** Refresh review of 7 paths covering the world-admin CLI (`daydream/admin.py`), LLM safety primitives (`daydream/llm/safety.py`), the DB-loaded data-skill loader/executor (`daydream/skills/data.py`), the new `description` migration (`migrations/005_skills_description.sql`), and the three companion test files. Scope overlaps significantly with the earlier same-day review (paths, commit 50310c2); this refresh picks up the `c33fdaa` hardening commit that (a) extended `data._narrative_text` to cover the `mood` field so a banned-category string in `set_mood` cannot reach `toons.mood` → SPA `${name} (${mood})` render, (b) replaced the case-sensitive `str.replace(_CLOSE_TAG, ...)` in `safety.wrap_player_input` with `_CLOSE_TAG_RE = re.compile(r"</\s*player_input\s*>", re.IGNORECASE)` so case-variant and whitespace-padded close attempts are neutralized, (c) persisted the author-file `description` field through `cmd_skill_add`'s INSERT / ON CONFLICT clauses into the new `skills.description` column added by migration 005. All three changes tighten the existing safety surface — the prior review already verified the originals were secure; the deltas are defense-in-depth improvements. Migration 005 is a single static DDL (`ALTER TABLE skills ADD COLUMN description TEXT`) with no user input, so no injection surface. The author-file `description` value flows operator-authored → DB → `SkillSpec.description` → `prompts.interpreter_user` as `f"- {s.name}: {s.description}"` into the interpreter LLM prompt; this is an operator trust boundary (the operator runs `bin/game world skill add <file>` on a JSON file they control), the same category as any other operator-authored LLM prompt content. `_skill_validate` requires `description` to be a non-empty string and `.strip()`s it before storage. `cmd_skill_add` is CLI-only — not imported by `daydream/server.py` or `daydream/api/` — so the code path is unreachable from the web layer (re-confirmed this round). All SQL in scope is parameterized (`admin.cmd_skill_add` INSERT + ON CONFLICT, `admin.cmd_delete` cascade, `data._rows` SELECT, `_parse_pair` dict reads). Jinja `SandboxedEnvironment` (Jinja2 3.1.6, which includes fixes for CVE-2024-56201 and CVE-2025-27516) still guards authored templates; `{{ player_input }}` substitution injects a pre-computed string value, not template source, so a player cannot inject Jinja syntax. `safety.first_banned` regex patterns use `\b(?:...)\b` alternations without nested quantifiers, no ReDoS backtracking. Git history for the seven files shows no secrets and no prior-state sensitive content; the `-S "password|api_key|secret=|token="` search across all four non-test files returned nothing. The `_narrative_text` coverage set `("text", "seed", "name", "mood")` matches the fields the effect handlers actually persist (`_apply_narrate.text`, `_apply_add_item.name/seed`, `_apply_set_mood.mood`); extra fields the LLM might emit are dropped by the handlers. ### Accepted Risks

- Cookie `https_only=False` in `daydream/server.py:38` is documented inline ("friend-scope; box is on a private LAN/Tailscale only").
- No CSRF token on `/api/login` and `/api/logout`. `SessionMiddleware` defaults to `SameSite=Lax`, which blocks cross-site POST from carrying the session cookie; the worst-case residual impact under the documented threat model is forced-logout from a same-site phishing page, consistent with friend-scope posture.
- The 100.64.0.0/10 hardcoding in `daydream/api/access.py:25` is correct because Tailscale's CGNAT range is fixed by their design. A self-hosted Headscale with a custom range would need that constant updated. Documented in CLAUDE.md.
- `AccessMiddleware` reads `scope["client"][0]` directly rather than honoring forwarded-for headers. This is the secure default given uvicorn is not started with `--proxy-headers`. Operators who later put the app behind a reverse proxy must add `--forwarded-allow-ips` themselves AND audit that the proxy strips inbound `X-Forwarded-For`.
- `bin/game` sources `.env` and `~/.config/daydream/secrets.env` via `set -a; source`; a writable env file is equivalent to arbitrary code execution. Operator-controlled, gitignored, standard practice.
- `bin/game cmd_logs` passes `$1` through `_log_file` without validation; an operator typing `bin/game logs ../../etc/passwd` would resolve into `$RUNDIR/../../etc/passwd.log`. Operator-invoked only; anything the operator can `tail` through this path they can already read directly on the box.
- `bin/qpeek-bootstrap` clones `https://github.com/peterzat/qpeek` and runs `pip install -e` against it. Supply-chain dependency on the project owner's own GitHub account; same trust boundary as the rest of the dev toolchain.
- `/cache/{world}/{target_kind}/{target_id}/{filename}` in `daydream/server.py:91` is intentionally unauthenticated so pre-auth `<img src>` fetches render. `AccessMiddleware` is the real gate; segment validation blocks traversal.
- Tailscale-mode `POST /api/login` at `daydream/api/auth.py:32-34` short-circuits to `authed=True` + redirect without checking the password. Deliberate: `AccessMiddleware` has already CGNAT-gated the request, so the password check would be redundant ceremony. Documented in the module docstring and in CLAUDE.md "Auth" + "Network access" sections.
- Data-skill `set_mood` effect handler (`daydream/skills/effects.py:128-151`) accepts an LLM-chosen `toon_id` without scoping it to the actor's world / room / ownership. In v0 single-player + friend-scope this sits inside the existing trust boundary — an authenticated player can already mutate state through play; the LLM is not a distinct actor against them. v2's per-effect jsonschema + target-authorization pipeline is tracked under BACKLOG `skills-authoring-and-security`. Documented in `daydream/skills/effects.py` module docstring.
