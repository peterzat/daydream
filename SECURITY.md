**Summary:** Path-scoped review of the toon-slot-management landing (commit `63e1ed0`). Five files reviewed: `daydream/api/slots.py` (new, four endpoints), `daydream/api/auth.py`, `daydream/api/ws.py`, `daydream/toons.py`, `daydream/server.py`. The new attack surface is the four slot endpoints (`GET /api/slots`, `POST /api/slots/{slot}/{create,claim,kick}`) and a session-UUID stamper (`auth._ensure_session_id`) used to bind a controller identity to each session. Auth posture: every endpoint goes through `_require_authed` which mirrors the existing `is_authed` policy (tailscale mode â†’ unconditional pass; public mode â†’ session-cookie check). All four endpoints sit behind `AccessMiddleware` at the outer edge, so non-tailnet clients are rejected with 403 / WS close 1008 before any of this code runs in default mode. v1 friend-scope is documented in the `slots.py` module docstring: any auth'd session can create / claim / kick any slot â€” per-session ownership lands with v2. SQL: every helper in `toons.py` (`get_toon_by_session`, `get_human_slots`, `_slot_occupied`, `create_toon_in_slot`, `claim_slot`, `kick_slot`) uses parameterized queries; user-supplied `name` and `appearance_seed` are bound parameters, never interpolated into SQL strings. Input validation: `create_slot` rejects non-dict bodies (400), non-string / whitespace-only `name` or `appearance_seed` (400), and out-of-range slots (404); `_validate_slot` constrains the path-segment integer to `1..5` via `HUMAN_SLOT_RANGE`. The session id stamped by `_ensure_session_id` is a `uuid.uuid4()` (cryptographically random per Python's `uuid` module on Linux), not a guessable counter. XSS surface for attacker-controlled `name` and `appearance_seed`: the SPA renders toon names via `escape()` (HTML-entity encoding for `&<>"`) in the slot panel innerHTML paths (`web/assets/main.js:211, 218, 225`), via `textContent` for the room toons sidebar (`main.js:37`), and via `escape()` for the `say`-event speaker name (`main.js:89`). The `examine` core skill reflects `toon.appearance_seed` and `toon.seed` into a `narrate` event (`daydream/skills/core.py:74`); the SPA renders narrate via `textContent` (`main.js:93`) so reflected-XSS is not exploitable through that path. LLM exposure for attacker-controlled toon strings: data-skill template rendering only sees `player_input`, `actor_id`, `room_id`, and `memories` (`daydream/skills/data.py:291-297`); the toon's `name` and `appearance_seed` do not flow into the LLM prompt, so prompt-injection via toon-name is not a vector. Session-id stamping in tailscale mode is intentionally permissive: a tailnet GET to `/api/slots` (or POST to any slot endpoint) gets a fresh `Set-Cookie` on first hit because `is_authed` returns True in tailscale mode and `_ensure_session_id` mutates `request.session`; this is the same trust class as the prior-accepted "Tailscale-mode `/api/login` short-circuit" risk and is consistent with the documented threat model. The `claim_slot` / `kick_slot` v1 friend-scope (no per-session ownership check) is explicit in the module docstring and in the spec's "v1 friend-scope" callout; v2 multi-user-shared-world tightens it. Git history (`git log -p --follow`) for all five files shows no introduced or removed secret values; the only `password` / `secret` / `token` occurrences in the diffs are documentation of the prior-accepted Tailscale-mode skip and the `DAYDREAM_PASSWORD` / `DAYDREAM_SESSION_SECRET` env-var contract. No PII beyond commit-author metadata (out of scope). The TOCTOU between `_slot_occupied` and the subsequent INSERT in `create_toon_in_slot` is a correctness / spec-conformance concern (the spec promises "one 200, one 409, never both 200" under concurrent creates, and the absence of a `(slot, world_id) WHERE kicked_at IS NULL` partial unique index means SQLite's single-writer serialization is the only thing preventing a double-insert) â€” not a security issue, since two duplicate slot rows do not bypass auth, escalate privileges, or expose data; flagged here for awareness but belongs in CODEREVIEW, not SECURITY. Request-body size is unbounded (no `client_max_body_size` / Starlette body-size cap), which lets a tailnet member POST an arbitrarily large `name` or `appearance_seed` and persist it; this is the same trust class as the documented friend-scope "any auth'd session can do everything" posture and is not a security finding under that threat model.

The previously-accepted WARN for stored prompt-injection via captured memory text is unchanged by this turn â€” the new endpoints do not write to or read from the memory pipeline. Per skill instructions, no accepted risks are re-flagged here.

### Accepted Risks

Carried forward from prior reviews; none re-flagged this scope:

- Cookie `https_only=False` in `daydream/server.py:42` (friend-scope; LAN/Tailscale only).
- No CSRF token on `/api/login` and `/api/logout`. `SameSite=Lax` blocks cross-site POST from carrying the session cookie; worst case under the documented threat model is forced-logout from a same-site phishing page. The new `POST /api/slots/{slot}/{create,claim,kick}` endpoints inherit the same posture (no CSRF token); same-origin-only state changes from a tailnet-resident attacker page are blocked by `SameSite=Lax` for the `daydream_session` cookie, and unauthed cross-site POSTs from outside the tailnet are 403'd by `AccessMiddleware` before reaching these handlers. Multi-user differentiation that would tighten this lands with v2.
- The `100.64.0.0/10` Tailscale CGNAT hardcoding in `daydream/api/access.py`. Self-hosted Headscale with a custom range would need this updated. Documented in CLAUDE.md.
- `AccessMiddleware` reads `scope["client"][0]` directly rather than honoring forwarded-for headers. Secure default given uvicorn does not start with `--proxy-headers`. Operators putting the app behind a reverse proxy must add `--forwarded-allow-ips` AND audit that the proxy strips inbound `X-Forwarded-For`.
- `bin/game` sources `.env` and `~/.config/daydream/secrets.env` via `set -a; source`; a writable env file is equivalent to arbiter code execution. Operator-controlled, gitignored, standard practice. The `DAYDREAM_DRIFT_*` and `DAYDREAM_MEMORY_*` env vars sit inside this same trust boundary; their parsers float-/int-coerce with a `ValueError` fallback to the default and never propagate to security-sensitive code paths.
- `bin/game cmd_logs` passes `$1` through `_log_file` without validation. Operator-invoked only.
- `bin/qpeek-bootstrap` clones from `github.com/peterzat/qpeek` and runs `pip install -e`. Same trust boundary as the rest of the dev toolchain.
- `/cache/{world}/{target_kind}/{target_id}/{filename}` in `daydream/server.py:118-130` is intentionally unauthenticated so pre-auth `<img src>` fetches render. `AccessMiddleware` is the real gate; segment validation blocks traversal.
- Tailscale-mode `POST /api/login` at `daydream/api/auth.py:46-50` short-circuits to `authed=True` without checking the password. `AccessMiddleware` has already CGNAT-gated the request. Documented in module docstring + CLAUDE.md.
- Data-skill `set_mood` effect handler accepts an LLM-chosen `toon_id` without scoping it (`daydream/skills/effects.py:128-151`). v2's per-effect jsonschema + target-authorization pipeline is tracked under BACKLOG `skills-authoring-and-security`. The `daydream.toons.set_mood` helper itself is a parameterized UPDATE; the unscoped-target risk lives at the effect-handler call site, not in the helper.
- `bin/vllm-bootstrap` and `bin/memory-bootstrap` interpolate `$MODEL` into an unquoted heredoc that becomes Python source. Same operator-trust class as `bin/game`'s env sourcing.
- Stored prompt-injection via captured memory text bypasses `wrap_player_input` containment (`daydream/skills/data.py:362`, `daydream/memories.py:175-209`, `skills/{rook,iris}.json:6`, plus the parallel drift instance at `daydream/drift.py:179-185`). Capture-time banlist is in place; `<memory>...</memory>` separator wraps the rendered text; `</memory>` substrings are NOT neutralized at retrieval time. Realistic impact at v0 is bounded to mild voice/tone deviation, not auth bypass / RCE / data exfiltration. Deeper fix is v2's `skills-authoring-and-security`.
- `/status/drift` in `daydream/server.py:60-80` is unauthenticated by design (no session check; only `AccessMiddleware` gates it). Returns three integer counters with no PII or secrets; in tailscale mode it is tailnet-only, in public mode it exposes "drift has fired N times" to any reader. Same trust class as `/login` and `/cache/...`.
- v1 friend-scope on the slot endpoints (`daydream/api/slots.py`): any auth'd session can create / claim / kick any slot â€” there is no per-session ownership enforcement on `claim_slot` or `kick_slot`. Explicit in the module docstring and in SPEC 2026-05-07's v1 friend-scope callout; v2 multi-user-shared-world introduces the per-session ownership check.
- `_ensure_session_id` (`daydream/api/auth.py:29-39`) stamps a fresh `uuid.uuid4()` into `request.session` on first read in tailscale mode, even on slot-endpoint hits that did not go through `POST /api/login`. Consistent with the documented "tailnet membership IS the auth" posture; the stamped UUID is the slot picker's controller identity, not an authentication token.
- Request body size on `POST /api/slots/{slot}/create` is unbounded (no `client_max_body_size` / Starlette body-size cap). A tailnet member can POST an arbitrarily large `name` / `appearance_seed` and persist it. Same trust class as the documented friend-scope "any auth'd session can do everything" posture.
