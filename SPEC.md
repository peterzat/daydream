## Spec — 2026-06-30 — verification infrastructure: batch and automate the eyeballs

**Goal:** Convert the project's qualitative "eyeball" checks into ratify-once-then-mechanical regression proxies, and batch the irreducible human glances into a single artifact, so future autonomy spends precious eyes-on time on gameplay feel rather than on mechanical "does it render / does the voice read right" verification. This generalizes three already-deferred backlog items (`forge-render-drift-anchor`, `claude-vision-quality-gate`, `voice-baseline-add-model-helper`) and the pending manual eyeballs from the playtest-polish spec into one coherent verification stack.

### Acceptance Criteria

*Mechanical proxies for image aesthetics*

- [ ] **The authored forge render gains a mechanical regression proxy.** The `r-forge` seed (from `worlds/bunny.json`) is a 4th aesthetic anchor at `tests/drift/aesthetics/forge.json`, picked up automatically by the existing perceptual-hash framework (`tests/drift/test_image_perceptual.py` globs `aesthetics/*.json`); a committed `tests/baselines/image_forge.golden.json` baseline exists so a forge-render regression (LoRA/checkpoint/workflow/seed change shifting the dHash past tolerance) fails `tier_long` by design. Verifiable: the corpus loader includes `forge`; the perceptual probe renders + compares against the golden under real GPU. **(Anchor + framework shipped and committed; the first `image_forge.golden.json` is the one remaining operator action — render via `bin/game review`, confirm the forge reads right, then `mv` the latest to golden. This is the only unmet box.)** Closes `forge-render-drift-anchor`.

- [x] **An automated aesthetic-conformance gate exists, off by default.** A Claude-vision rubric gate (`daydream/testing/vision_gate.py`) rates a rendered image against the WHIMSY rubric via one Opus vision call and returns a structured `{score, pass, reason}`; it is design-time/test-time ONLY (never called by the running game) and OFF unless `DAYDREAM_CLAUDE_VISION_GATE` is set, so routine and autonomous runs cost nothing and add no runtime cloud dependency (consistent with the local-only runtime generation policy). A `tier_long` probe (`tests/drift/test_image_claude_vision.py`) asserts each anchor clears the threshold when enabled and SKIPS cleanly when not. Verifiable GPU/key-free: a mocked-litellm test asserts the rubric prompt carries the WHIMSY constraints, a below-threshold response fails, and the probe is skipped when the flag is unset. Closes `claude-vision-quality-gate`.

*Batched human review (decoupled from the live server)*

- [x] **One command produces one artifact for the whole qualitative review, with no live server and no live-state mutation.** `bin/game review` loads `worlds/bunny.json` into a throwaway temp DB (never touching `~/data/daydream/worlds-dev/live.db`, never requiring `bin/game up`), renders every aesthetic anchor (including the forge) and generates a fresh `talk` voice sample for each authored NPC (Rook/Iris/Bram) from their shipped `worlds/bunny.json` dialogue prompts, and emits a single self-contained `index.html` contact sheet referencing each render and each sample, every item labelled with its source seed/prompt (and, when the vision gate is enabled, its rubric verdict). The artifact lands under `~/data/daydream/reviews/<ts>/` (ephemeral) with a `pretty`-able path into `docs/pretty/` for keepers. Verifiable GPU/LLM-free: a mocked-engine test asserts the harness loads bunny.json into a temp DB and writes an `index.html` referencing each anchor and each NPC sample.

- [x] **The one irreducible browser check is batched into the same artifact, not a separate hunt.** The connection-state overlay's *behavior* (the single calm "sleeping" overlay, capped-backoff reconnect, the `world_changed` "shifts" beat, no growing disconnect-line pile) stays covered by the existing source-scan logic test in `tests/test_frontend.py` (extended if a gap is found); the irreducible *live visual* across a real `down`/`up` cycle and a `world swap` is captured as an explicit, minimal browser checklist section inside the review `index.html`, so it is glanced once alongside the renders and voices. No headless-browser dependency is added (documented as a deliberate out-of-scope call). Verifiable: the logic test passes; the review HTML contains the browser-checklist section.

*Auto-extending voice regression*

- [x] **Adding a voice baseline extends the regression set with no code edit.** `tests/test_voice_baseline.py` derives its parametrization from a manifest/front-matter convention over `docs/pretty/voice-samples/*.md`, so a new regression-tracked baseline (e.g. a future model A/B) auto-extends the tic-distinctness check, while the two in-tree documented-failure captures (Mistral-Nemo) are excluded by an explicit discriminator. Verifiable: a unit test of the discriminator (a tracked capture is included and asserts distinct openers; a failure-mode capture is excluded); the existing pre-fix/post-fix AWQ regression demonstration is preserved. Closes `voice-baseline-add-model-helper`.

*Cross-cutting*

- [x] **Tiers green; docs + backlog rolled forward.** `bin/game test short` and `bin/game test medium` exit 0 with paired GPU/LLM-free tests (mocked engines) for every non-GPU criterion above; `tier_long` gains the forge perceptual probe and the gated vision probe. README + CLAUDE.md + TESTING.md document `bin/game review` (the batched-eyeball workflow) and the vision gate; `BACKLOG.md` closes `forge-render-drift-anchor`, `claude-vision-quality-gate`, and `voice-baseline-add-model-helper`, and records the only remaining manual actions: the one-time forge-golden ratification + contact-sheet quality glance (one batched session) and the live-overlay browser check.

### Context

**Origin.** Operator directive (2026-06-30): "implement as much of the infrastructure as possible autonomously; the end state actually minimizes (or batches) eyeballs needed; for future autonomy the design needs to save precious eyes-on time for gameplay." Follows the playtest-polish spec (14/14), whose three flagged-manual eyeballs (connection overlay, NPC voices, forge image) plus a needed live reset are exactly the per-turn human cost this turn drives down.

**The eyeball-cost model (what we are minimizing).** Three kinds of per-turn human verification existed: (1) image aesthetics, (2) NPC voice, (3) the SPA connection overlay. Each previously meant a live reset + a browser session. This turn splits each into a *mechanical regression proxy* (caught automatically forever after one ratification) plus, where a render/voice genuinely needs a human, a *single batched glance* at one offline-produced artifact. The biggest lever is decoupling content verification from the live server: `bin/game review` renders the shipped world's content from a temp DB, so the operator never resets live or opens a browser just to confirm the forge looks like a forge or the voices read in character.

**Generation policy (load-bearing, CLAUDE.md).** Runtime stays local-only with no cloud key. The Claude-vision gate is design-time/test-time tooling invoked by the operator/agent, never by the running game, and is gated off by default; enabling it requires an `ANTHROPIC_API_KEY` in the dev environment, consistent with "design-time heavy generation by Opus is fine." Offline voice samples and image renders use the same local engines (vLLM/ComfyUI) as the game.

**Code anchors (verify before editing).** Perceptual framework: `tests/drift/conftest.py` (dhash, `load_aesthetics_corpus`, golden load/write), `tests/drift/test_image_perceptual.py` (parametrized probe), `tests/drift/aesthetics/*.json` (anchor shape), `tests/baselines/image_*.golden.json` (golden shape). Image gen: `daydream/images/client.py` (`generate_image`, `EphemeralTarget`, `WHIMSY_PROMPT_SUFFIX`, `_execute_workflow` mock point), `daydream/testing/human_eval.py` (the render-corpus pattern). Offline world: `daydream/llm/bootstrap.py:load_world`, `daydream/admin.py:main(["load", ...])`, `tests/test_world_load.py` (temp-DB pattern). Voice: `daydream/voice_samples.py` (hermetic dispatch), `daydream/skills/data.py:execute` (the dialogue call path), `worlds/bunny.json` (per-NPC dialogue prompts), `tests/test_voice_baseline.py` (tic-regression). Overlay: `web/index.html` (`#dream-overlay`), `web/assets/main.js` (`showDreamOverlay`/`hideDreamOverlay`, "the dream is sleeping"/"shifts", capped backoff), `tests/test_frontend.py`. Dispatch: `bin/game` (bash case table), `daydream/testing/__main__.py` (tier markers).

**zat.env practices.** Land as small, individually-tested, committable increments (suggested order: forge anchor, then the review harness, then the vision gate, then the voice manifest, then the overlay checklist, then docs). The real GPU render that produces `image_forge.golden.json` and the first real contact sheet is a single coordinated pass (server down for GPU safety, per the CLAUDE.md automation policy). Write paired tests in the same increment; mocked engines keep everything GPU/LLM/key-free in `tier_short`/`tier_medium`.

**Out of scope.** Headless-browser automation of the overlay (heavy dependency on a single-dev box; the logic is already source-scan-tested and the visual batches into the review sheet). Rendering every authored room (anchors + forge cover the aesthetic-conformance need this turn). A standing CI vision-gate budget (the gate is opt-in; wiring it into a paid CI loop is a later call). Any model swap or runtime cloud call.
